### **1. ISO/IEC 27017 (Cloud Security Guidelines)**

- **ISO**: International Organization for Standardization
- **IEC**: International Electrotechnical Commission

**Explanation**:  
ISO/IEC 27017 is a set of guidelines developed for cloud service providers and customers. It focuses on:

- Implementing **information security controls**.
- Addressing **cloud-specific risks**.
- Ensuring best practices for cloud service security, such as data segregation, virtual machine security, and more.


### **2. ISO/IEC 27018 (Data Protection in Cloud)**

- **PII**: Personally Identifiable Information

**Explanation**:  
ISO/IEC 27018 provides standards to **protect personal data** in cloud environments, especially public clouds. It ensures:

- **Confidentiality** of PII.
- **Transparency** in data processing.
- Control for individuals over their data.


### **3. NIST SP 800-53 and NIST SP 500-291**

- **NIST**: National Institute of Standards and Technology
- **SP**: Special Publication

**Explanation**:  
These are U.S. government-issued security guidelines:

- **NIST SP 800-53**: A comprehensive catalog of **security and privacy controls** for [[federal]] information systems. It addresses access control, audit, incident response, and more.
- **NIST SP 500-291**: Provides **specific cloud computing recommendations**, including guidance on architecture, security, and deployment models for cloud systems.


### **4. Cloud Security Alliance (CSA) Controls Matrix**

- **CSA**: Cloud Security Alliance
- **IAM**: Identity and Access Management

**Explanation**:  
The **CSA Controls Matrix** is a cybersecurity control [[framework]] tailored for cloud environments. It:

- Maps controls to **industry-accepted security standards**.
- Covers essential cloud domains like:
    - **IAM**: Managing user identities and access rights.
    - **Compliance**: Adhering to legal, regulatory, and policy requirements.
    - **Risk Management**: Identifying and mitigating cloud-related risks.

# Classless Inter  Domain Routing

- Old way to manage IP address is using Classful addressing
- In cloud we use CIDR for managing IP addresses.
- There is no class so even if CIDR addresses are written with IPV4 address followed by slash and then followed by subnet mask but still there will be no class in IPV4 portion.

![[Pasted image 20250719081406.png]]
- Subnet mask represent Network bits.
- You can calculate host bits by subtracting subnet mask from total bits of 32.

![[Pasted image 20250719081001.png]]
- First 3 octets defines a unique network and with 4th octet we can have like 254 valid hosts and _ . _ . _ . 0 is reserved for Network ID , to connect to network and  _ . _ . _ . 255 is reserved for broadcast , in case external party wants to send a message to all the host of the network.


# How to do subnetting

- Now we need to configure whole network.
- It is done so that some of our resources can be kept public and some other resources like database can be kept private.
- First thing to do is to find how many IP addresses are required to do such configuration.
- Suppose a company who wants to do subnetting says it might require them 100 devices to provide public services and to manage database and operations will also require 100 devices. So in total they require 200 IP or 200 hosts.
- Now cloud provider tries to take enough IP addresses to accommodate 200 hosts. Like 256 is near about , 128 will be too less and 512 will be to much.
- Don't touch network bits while doing subnetting which is denoted by subnet bits.
- We can do that by reserving one bit , like if we fix MSB as 0 then rest 7 bits will make up 128 IP's so network is divided into half similarly we can set MSB as 1 for other half.
- Now  create masked address 255.255.255.128.  As we have fixed one bit so subnet mask will become /25. Any address trying to access our website will perform AND operation with masked address which will tell that in which half of network it belongs either .0 or .128.

# Ports

- They are Virtual Endpoints. So , when you search for a website then the request goes to DNS server which provides the IP address of that website , based on which we can locate the network and further locate the server. But a single server provide various services like website service , mail service  etc. so now how to find the service that we want , so all those services will have different port no. and based on that we can connect to correct service. 
- Like if your URL has 80 port no. (HTTP) like `128.10.10.92:80` then it will tell the server that you want to access the website.
- Like if you want to access mail service then use port no. 25 (SMTP) like `128.10.10.92:25`.


![[Pasted image 20250721083533.png|400]]
- Example: 25(SMTP), 80(HTTP), 443(HTTPS) , SSH(22), FTP (20,21)
- Registered Ports: Many developers and 3rd party companies like Adobe, Oracle, SQL Server , NodeJS, MySQL etc. uses registered ports as per their requirements.
- If you are trying to access frontend of a server then in case of React application Port no. 3000 will be used, in case of backend if it is running on NodeJS then Port no. 5000 will be used
- Dynamic / Private Ports: These ports are selected from range 49152 - 65535 by your system's OS. So that server is also able to know the actual source of request.

# Serverless Architecture

- A serverless architecture relies on managed services where you only pay for what you use, without worrying about infrastructure.
- Major examples are Amazon Lambda , Google Cloud Functions.
- In serverless architecture , server does exist but we don't need to configure them like we do in server based architecture (like configuring EC2 instance).
- It is more cost effective as it is Pay as you Execute service. Your code will remain in passive state until it receives a request from client. But in server based architecture you will be charged per hour for the time your code is running on the server.
- In this architecture, control is with cloud provider  but in server based control control is with developer.
- First 1 Million requests are free and after that nominal 0.20 $ per request is charged.

# Kubernetes (K8s)
- It's an orchestration tool.
- It runs and manages all the containers.
- In CI/CD pipeline their is continuous flow of new changes over deployed code in dev ops so in that we require a tool which manages the deployment automatically.
- In Kubernetes Network their exists a master node and multiple worker nodes.
- Whole network including master and multiple worker nodes is called a cluster.
- It's platform as a Service

![[Pasted image 20250807082914.png|400]]
- API Server is like the reception room where all clients request arrives.
- After validating the request it is forwarded to ETCD which will check if their is availability of a `POD`
	![[Pasted image 20250807081747.png|300]]
	A pod can have single container or it can have multiple container.
	![[Pasted image 20250807081804.png|200]]
- If pod is available then Scheduler allocates pod as per request. Scheduler also saves in `ETCD` that which pod has been assigned for request.
- Now API Server knows the location on which pod to send the request, so it sends the request to `Kubelet` (like agent) of that pod. Kubelet now deploys the code changes on container in pod.
- In a pod their is element called `Kube-proxy` , it manages load balancing and networking by establishing  communication among pods.
- If their is any kind of error encountered like code fails, pod require some dependencies or our system should be scalable. So such operations are managed by control manager. 
- Different controllers are their like replication controller, deployment controller etc.
- Kubelet gives instruction to Container runtime , like how much time container should run and when to stop etc.
